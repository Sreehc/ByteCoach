## 二、整体架构设计（8.0 / 10）

### 亮点

- **前后端分离架构清晰**：Spring Boot 3 单体后端 + Vue 3 SPA 前端，职责划分明确，符合 MVP 阶段的务实选择。
- **后端模块化包结构优秀**：按业务域划分 `auth`、`chat`、`knowledge`、`interview`、`plan`、`wrong`、`dashboard`、`category`、`question`、`user` 等模块，每个模块内部遵循 `controller → service → mapper → entity → dto/vo` 的标准分层。
- **AI 层抽象合理**：`LlmGateway`（接口）→ `OpenAiCompatibleLlmGateway`（实现）→ `AiOrchestratorService`（编排）→ `PromptTemplateService`（模板）四层解耦，方便后续切换 LLM 提供商。
- **知识检索与 AI 编排解耦**：`KnowledgeRetrievalService` 独立于 AI 模块，Chat 模块通过 `AiOrchestratorService` 组合调用，依赖方向正确。
- **前端路由设计合理**：`AppLayout` 嵌套路由 + `pageMeta` 集中管理页面元信息，`guestOnly` / `requiresAuth` 守卫逻辑简洁有效。

### 问题

- **单体内缺少模块边界约束**：所有业务模块共享同一个 Spring 上下文，模块间可以直接注入对方的 Service，长期可能导致耦合加重。建议通过 `package-private` 可见性或 ArchUnit 测试加以约束。
- **缺少 CQRS 或读写分离思路**：Dashboard 的聚合查询（`DashboardMetricsMapper`）与业务写入共用同一数据源，高并发场景下可能产生性能瓶颈。
- **Redis 引入但未使用**：`pom.xml` 中包含 `spring-boot-starter-data-redis`，`application-dev.yml` 配置了 Redis 连接，但代码中没有任何 Redis 操作。JWT 黑名单、会话缓存、限流等场景均未落地。

## 三、后端代码质量（7.0 / 10）

### 亮点

- **统一响应封装**：`Result<T>` + `ResultCode` 枚举，全接口返回格式一致。
- **全局异常处理完善**：`GlobalExceptionHandler` 覆盖了 `BusinessException`、`MethodArgumentNotValidException`、`BindException`、`ConstraintViolationException`、`HttpMessageNotReadableException` 和兜底 `Exception`，异常分层合理。
- **JWT 实现规范**：使用 `jjwt 0.12.6`，支持 Base64 和 UTF-8 密钥自动判断，`JwtAuthenticationFilter` 继承 `OncePerRequestFilter`，过滤逻辑简洁。
- **MyBatis-Plus 使用规范**：`BaseEntity` 统一 `id`（雪花算法）、`createTime`、`updateTime` 自动填充；Service 层继承 `ServiceImpl`，减少模板代码。
- **Dashboard SQL 质量高**：`DashboardMetricsMapper` 使用注解 SQL，`COALESCE`、`NULLIF`、`LEFT JOIN` 处理空值边界，聚合逻辑清晰。
- **知识导入与切分逻辑完整**：`KnowledgeServiceImpl.importSeed()` 实现了幂等导入（通过 `fileUrl` 判断是否已存在）、`forceRebuild` 强制重建、Markdown 按标题段落切分、token 估算等。

### 问题

- **多个 Controller 仍为硬编码占位实现**：
  - `InterviewController`：`start()`、`current()`、`detail()` 全部返回写死的假数据，未接入数据库。
  - `WrongController`：`list()`、`detail()`、`updateMastery()`、`delete()` 全部为硬编码返回，无任何数据库操作。
  - `PlanController`：`generate()`、`current()`、`tasks()`、`updateTaskStatus()` 同样为硬编码返回。
  - 这三个模块是核心业务链路的关键环节，占位实现直接导致前端页面无法展示真实数据。

- **`AiOrchestratorService.scoreInterviewAnswer()` 返回固定分数**：无论用户回答什么，都返回 `78` 分和固定的 `standardAnswer`、`followUp`，AI 评分逻辑未真正落地。

- **`KnowledgeRetrievalServiceImpl` 检索算法过于简单**：当前实现是加载全部 chunk 到内存后逐条做 `String.indexOf()` 关键词匹配，数据量增长后会有严重性能问题。且不支持语义检索，检索质量有限。

- **`UserServiceImpl.createUser()` 缺少事务注解**：虽然单表操作通常不需要，但如果后续扩展为多表写入（如初始化用户配置），缺少 `@Transactional` 会有隐患。

- **`ChatServiceImpl.send()` 中 LLM 调用在事务内**：`@Transactional` 包裹了整个 `send()` 方法，其中包含 HTTP 调用 LLM 的远程请求，会导致数据库连接长时间被占用。建议将事务范围缩小到数据库操作部分。

- **缺少分页**：所有列表接口（`/chat/sessions`、`/knowledge/list`、`/question/list`、`/wrong/list`）均返回全量数据，未使用 MyBatis-Plus 的 `Page` 分页能力。

- **缺少参数校验注解**：`ChatSendRequest`、`InterviewStartRequest` 等 DTO 使用了 `@Valid`，但很多字段缺少 `@NotBlank`、`@Size` 等约束注解。

- **`CorsConfig` 允许所有来源**：`addAllowedOriginPattern("*")` + `setAllowCredentials(true)` 在开发环境可以接受，但生产环境必须收紧。

## 四、前端代码质量（7.5 / 10）

### 亮点

- **TypeScript 全面使用**：所有 `.ts`、`.vue` 文件均使用 TypeScript，`types/api.ts` 集中定义了所有接口类型，类型覆盖完整。
- **API 层设计清晰**：每个业务模块独立一个 API 文件（`auth.ts`、`chat.ts`、`dashboard.ts` 等），统一使用 `request<T>()` 封装，类型安全。
- **HTTP 拦截器逻辑完善**：请求拦截器自动注入 Token，响应拦截器处理 401 自动跳转登录、统一错误提示。
- **Pinia Store 设计合理**：`useAuthStore` 使用 Composition API 风格，`persist()` / `clear()` 封装 Token 和用户信息的存取，`restoreProfile()` 支持页面刷新后恢复登录态。
- **路由守卫设计巧妙**：`restored` 标志避免重复调用 `restoreProfile()`，`guestOnly` / `requiresAuth` 双向守卫防止已登录用户访问登录页。
- **Dashboard 页面体验优秀**：首次访问引导卡片（`showGuideCard`）、骨架屏加载态、空状态提示、快捷入口、进度条等交互细节完整。
- **Chat 页面功能完整**：会话列表 CRUD、消息历史加载、Markdown 渲染、RAG 引用展示、loading 态均已实现。
- **Admin 页面功能丰富**：分类管理、题库 CRUD、知识文档导入/重切分/重建索引三个 Tab 完整实现。

### 问题

- **Interview、Wrong、Plan 三个页面为纯静态占位**：
  - `InterviewPage.vue`：无任何 `<script setup>` 逻辑，未接入 `/api/interview` 接口，展示的是硬编码的假数据。
  - `WrongPage.vue`：`wrongItems` 为本地硬编码数组，未调用任何 API。
  - `PlanPage.vue`：`tasks` 为本地硬编码数组，未调用任何 API。
  - 这三个页面与后端占位 Controller 形成"双端占位"状态，用户无法走通完整业务闭环。

- **Chat 页面 Markdown 渲染存在 XSS 风险**：使用 `v-html="renderMarkdown(message.content)"` 直接渲染 LLM 返回的内容，未做 DOMPurify 消毒。恶意构造的 LLM 输出（如 prompt injection）可能注入脚本。

- **`storage.ts` 使用 `localStorage`**：Token 存储在 `localStorage` 中，存在 XSS 攻击下的 Token 泄露风险。更安全的做法是使用 `httpOnly` Cookie。不过对于 MVP 阶段，这是可接受的权衡。

- **前端 `ResultCode` 不匹配**：`http.ts` 中判断 `payload.code !== 200` 为成功，但后端 `ResultCode.SUCCESS` 的 code 值需要确认是否为 `200`。如果不一致，所有接口都会被误判为失败。

- **缺少前端测试**：无任何单元测试、组件测试或 E2E 测试。

- **未配置 Vite 代理**：`vite.config.ts` 中没有配置 `server.proxy`，前端直接请求 `http://localhost:8080/api`，依赖 CORS 放行。配置代理可以避免 CORS 问题并简化开发体验。

- **`marked` 库版本过新**：`package.json` 中 `"marked": "^18.0.2"`，`marked` 18.x 是 2025 年的版本，API 可能有 breaking changes。建议锁定版本或确认兼容性。

## 五、数据库设计（7.5 / 10）

### 亮点

- **表结构设计规范**：11 张表覆盖了用户、分类、题目、知识文档、知识分片、聊天会话、聊天消息、面试会话、面试记录、错题、学习计划、计划任务等核心实体。
- **字段类型选择合理**：`BIGINT` 主键、`TEXT` 大文本、`JSON` 引用数据、`DECIMAL(5,2)` 分数、`TINYINT` 状态标志等，类型匹配业务语义。
- **索引设计到位**：所有外键字段均有索引（`idx_chat_session_user_id`、`idx_interview_record_session_id` 等），高频查询字段覆盖良好。
- **CHECK 约束使用得当**：`category.type`、`chat_session.mode`、`wrong_question.source_type` 均有 CHECK 约束，保证数据完整性。
- **初始化数据完整**：`init.sql` 包含 demo 用户（BCrypt 密码）、示例分类、示例题目，可直接跑通演示。

### 问题

- **缺少外键约束**：所有表之间的关联仅通过 `BIGINT` 字段维系，没有 `FOREIGN KEY` 约束。例如 `chat_message.session_id` 引用 `chat_session.id`，但没有外键约束，删除 session 后可能留下孤立 message。（注：MyBatis-Plus 项目通常不使用物理外键，这是设计权衡，但仍需注意数据一致性。）

- **`user.id` 使用 BIGINT 但 init.sql 中硬编码为 1**：MyBatis-Plus 的 `ASSIGN_ID` 会生成雪花 ID（19 位数字），但初始化数据的 ID 为 `1`，两者不一致。如果系统先用雪花 ID 创建用户，再执行 init.sql，可能出现 ID 冲突或不匹配。

- **缺少软删除**：所有表都是物理删除，`DELETE` 操作不可恢复。对于错题、学习计划等业务数据，建议增加 `deleted` 字段或使用逻辑删除。

- **缺少审计字段**：只有 `create_time` 和 `update_time`，没有 `created_by`、`updated_by` 字段，无法追溯操作人。

- **`wrong_question` 缺少唯一约束**：同一用户对同一道题可能重复录入错题，建议增加 `UNIQUE(user_id, question_id)` 约束。

- **`study_plan_task.status` 缺少 CHECK 约束**：与 `wrong_question.mastery_level` 类似，`status` 字段应该约束为 `'todo'`、`'done'` 等合法值。

## 六、安全性（6.0 / 10）

### 亮点

- **Spring Security + JWT 无状态认证**：配置规范，`SecurityFilterChain` 明确了白名单、ADMIN 权限、OPTIONS 放行等规则。
- **密码使用 BCrypt 加密**：`PasswordEncoderConfig` 配置了 `BCryptPasswordEncoder`，init.sql 中的密码也是 BCrypt 哈希。
- **环境变量管理敏感配置**：JWT Secret、数据库密码、LLM API Key 等均支持环境变量覆盖，避免硬编码。
- **白名单最小化**：`SecurityConstants.WHITE_LIST` 仅包含注册、登录、ping 和 Swagger 文档路径。

### 问题

- **JWT 登出为 no-op**：`AuthServiceImpl.logout()` 是空实现，Token 在有效期内仍然可用。应使用 Redis 黑名单机制，在登出时将 Token 加入黑名单，`JwtAuthenticationFilter` 中增加黑名单检查。

- **CORS 配置过于宽松**：`addAllowedOriginPattern("*")` + `setAllowCredentials(true)` 允许任意来源携带凭证发起请求，生产环境中必须限制为具体域名。

- **JWT Secret 默认值不安全**：`application-dev.yml` 中 `jwt-secret` 默认值为 `ByteCoachJwtSecretByteCoachJwtSecret123456`，如果生产环境忘记修改，将导致 Token 可被伪造。建议启动时校验 Secret 长度和复杂度。

- **`/api/admin/**` 仅检查角色，无接口级别的细粒度权限控制**：所有 ADMIN 接口共享同一个角色检查，无法区分"题库管理员"和"系统管理员"等更细粒度的权限。

- **缺少接口限流**：无 Rate Limiting 机制，暴力破解登录接口、恶意刷接口等场景没有防护。建议引入 Bucket4j 或 Spring Cloud Gateway 的限流能力。

- **Chat 页面 `v-html` XSS 风险**：如前所述，LLM 返回内容直接通过 `v-html` 渲染，存在 XSS 攻击面。

- **SQL 注入风险低但值得确认**：MyBatis-Plus 的 LambdaQueryWrapper 使用参数化查询，DashboardMetricsMapper 使用 `#{}` 占位符，SQL 注入风险低。但应避免任何 `${}` 拼接。

## 七、UI/UX 设计（8.5 / 10）

### 亮点

- **设计系统完整且有质感**：CSS 自定义属性（`--bc-ink`、`--bc-accent`、`--bc-surface` 等）构成统一的设计语言，`paper-panel`、`metric-card`、`surface-card`、`hard-button` 等组件类名语义清晰。
- **微交互动效细腻**：`metric-card:hover` 的背景色和阴影变化、`hard-button:active` 的 translateY 和 scale、`160ms cubic-bezier(0.16, 1, 0.3, 1)` 缓动曲线，都体现了对细节的关注。
- **Typography 层次分明**：`section-kicker`（小号大写标签）→ `page-title`（大号标题）→ `page-subtitle`（描述文本）→ 正文，视觉层次清晰。
- **空状态设计完整**：Dashboard 首次访问引导、Chat 空会话提示、Knowledge 空检索结果、Interview/Wrong/Plan 的空状态占位，每个页面都有友好的空状态引导。
- **加载态设计一致**：Dashboard 使用骨架屏（`animate-pulse`），Chat 使用加载占位块，Admin 使用按钮 loading 状态，加载体验统一。
- **配色方案专业**：深蓝主色调（`#2f4f9d`）搭配柔和的灰白背景，既专业又不沉闷，符合教育/工具类产品调性。
- **`prefers-reduced-motion` 适配**：对动画敏感的用户自动禁用动效，体现了无障碍设计意识。
- **`focus-visible` 焦点样式**：键盘导航时显示蓝色焦点环，提升了可访问性。

### 问题

- **不支持移动端响应式**：所有页面使用 `xl:grid-cols-*` 和 `lg:grid-cols-*` 做大屏适配，但没有针对手机屏幕（< 768px）的布局优化。侧边栏在小屏幕上不会折叠，内容可能溢出。

- **无深色模式**：所有颜色值固定为浅色主题，没有 `prefers-color-scheme: dark` 适配。对于长时间使用的面试准备工具，深色模式是刚需。

- **Element Plus 样式覆盖较多**：`main.css` 中大量使用 `!important` 覆盖 Element Plus 默认样式（如 `.el-input__wrapper`、`.el-button`），增加了维护成本。建议通过 Element Plus 的 CSS 变量或主题配置来定制，减少 `!important`。

- **中英文混排风格不统一**：页面中 kicker 使用英文（"Learning Overview"、"Guided Q&A"），正文使用中文，标题中英文混杂。建议统一语言策略，或明确这是有意为之的设计决策。

- **NavRail 无折叠能力**：左侧导航栏固定 300px 宽度，在内容区域较窄时会挤压主内容区。建议支持折叠为图标模式。

- **Chat 页面的消息气泡样式可以优化**：用户消息使用蓝色背景白色文字，但没有消息气泡的圆角指向箭头，视觉上更像是卡片而非对话气泡。

### 功能闭环评估

README 中声明的一期冻结范围是"登录 -> 问答/面试 -> 错题 -> 学习计划 -> Dashboard 的最小可展示闭环"。当前实际状态：

- 登录 ✅ → 问答 ✅ → 面试 ❌ → 错题 ❌ → 学习计划 ❌ → Dashboard ✅

**闭环在"面试"环节断裂**，后续的错题沉淀和学习计划生成均无法自然触发。用户只能使用 Chat 问答和 Dashboard 两个核心功能。

## 十、部署与运维（6.5 / 10）

### 亮点

- **docker-compose 提供本地依赖**：MySQL 8.4 + Redis 7.4 一键启动，init.sql 自动挂载到 `docker-entrypoint-initdb.d`。
- **环境变量配置完整**：后端所有敏感配置均支持环境变量覆盖，README 中列出了完整的变量清单。
- **Swagger/Knife4j API 文档**：自动生成 API 文档，便于调试和联调。

### 问题

- **无生产部署方案**：只有本地开发的 docker-compose，没有生产环境的 Dockerfile、K8s 配置、Nginx 配置等。
- **前端无环境区分**：只有 `.env.development`，没有 `.env.production`，生产环境的 API 地址需要额外配置。
- **日志配置基础**：`logback-spring.xml` 存在但未审查其内容，`application.yml` 中仅设置 `com.bytecoach: INFO`，缺少结构化日志和日志文件轮转配置。

- **`application.yml` 中 `log-impl: StdOutImpl`**：MyBatis-Plus 的 SQL 日志输出到标准输出，开发环境方便但生产环境应使用日志框架。

## 十二、关键问题汇总与优先级建议

### P0 - 必须修复（阻塞核心功能）

1. **Interview / Wrong / Plan 三个模块从占位实现改为真实业务逻辑**：这是 README 中声明的核心闭环，当前断裂导致产品无法演示完整价值。

### P1 - 强烈建议（影响安全性与可维护性）

## 十三、总结

ByteCoach 作为 MVP 阶段的项目，在架构设计、代码规范、UI 质量和文档完整性方面表现优秀，超出了多数同类项目在早期阶段的水准。后端的模块化分层、AI 服务抽象、Dashboard 的真实数据聚合，前端的设计系统、交互细节、TypeScript 类型覆盖，都体现了较高的工程素养。

当前最大的短板是**功能完整度**——Interview、Wrong、Plan 三个核心模块仍为占位实现，导致 README 中承诺的"学习闭环"在面试环节断裂。其次是**测试覆盖为零**和**安全细节**（JWT 登出、XSS、CORS）需要补齐。

建议按 P0 → P1 → P2 → P3 的优先级逐步推进，在补齐核心功能的同时，同步建立测试基线和安全防护，使项目从"可演示的骨架"进化为"可交付的 MVP"。
