# OfferPilot V2 差距清单与后续路线

> 本文档不再记录既往实施过程，也不再作为阶段执行备忘录使用。
>
> 当前正式产品说明已经回写到 `docs/prd / architecture / api / database / design / user-manual / deployment`。本文档只维护这些正式文档未覆盖的已知差距、后续修复方向与验收基线。

## 审视基线

- 审视日期：2026-05-14
- 对照范围：当前仓库代码、`sql/init.sql`、前端路由与页面、后端 controller/service、正式 docs
- 目标：明确哪些能力“已经可用但还没实现好”，以及下一轮应优先补哪些闭环

## 1. 产品与体验治理

### 1.1 现状诊断

当前系统的问题不是局部样式问题，也不是单个页面“看起来有点乱”的问题，而是信息架构、页面表达和文案口径同时失控。

现有问题已经形成系统性模式：

- 路由 `meta` 已经提供 `kicker + title + subtitle`
- `AppShellHeader` 再输出一层完整页头
- 页面正文继续补 hero 卡、说明卡、章节标题、辅助标签和摘要卡
- `shell-section-card`、`hard-chip`、`detail-pill`、`section-kicker` 被过度复用，导致每一块都像重点
- 多个页面首屏同时出现“标题 + 副标题 + 模块介绍 + 状态标签 + 指标卡 + 入口卡”
- 大量文案仍以系统视角表达，例如系统做了什么、模块是什么、当前接入了什么、后续会扩展什么

用户感知层面的直接问题已经很明确：

- 标题重复
- 副标题重复
- 同义说明重复
- 卡片过多导致注意力分散
- 首屏抓不到主动作
- 页面像产品说明书，不像可执行工具
- 文案像系统在自我介绍，不像在指导用户完成任务

### 1.2 问题分级

#### P0 结构性问题

- 重复页头
- 重复说明
- 页面首屏没有唯一主任务
- 信息卡片数量失控
- 全站文案视角错误

#### P1 表达性问题

- 术语重复
- 状态标签过多
- 按钮权重冲突
- 辅助信息抢主信息
- 空状态与引导语过于抽象

#### P2 视觉性问题

- 面板边框、阴影、色块和分组过密
- 多块卡片堆叠后形成视觉噪音
- 首屏密度过高，阅读路径不清晰

### 1.3 设计红线

以下内容不是建议，而是后续页面整改必须遵守的设计红线：

- 一个页面首屏只能有一个一级标题
- 一级标题下只能有一种说明来源，不能同时存在 `route subtitle + hero 描述 + 说明卡` 三套文案
- 非告警页面首屏禁止堆叠 3 个以上状态 chip
- 首屏禁止同时出现“主 hero + 功能导览卡 + 模块说明卡 + 指标卡”四层结构
- 一个内容页默认不超过 3 个首屏面板
- `section-kicker` 只能用于真正的章节分割，不能每张卡片都带
- `shell-section-card` 不能继续作为所有信息块的默认容器，必须区分主面板、次面板和信息条
- 首屏只能存在一个主 CTA，其它动作必须降级
- 同一屏内禁止出现两个以上视觉同权的主按钮

### 1.4 文案红线

以下内容是全站文案必须遵守的红线：

- 所有页面文案必须面向用户，不得面向系统
- 禁止把“模块定位、阶段进度、系统能力说明”写成页面主文案
- 禁止使用“当前已接入 / 当前预留 / 已切换为 / 后续将 / 先搭建入口”这类研发视角句式作为页面主表达
- 页面标题必须表达用户任务，不表达系统结构
- 副标题必须回答“你现在可以做什么”，不回答“这个模块是什么”
- 空状态文案必须给出下一步动作，不得只做介绍
- CTA 文案必须是用户动作，例如“开始面试 / 上传简历 / 记录投递 / 继续刷题”，不得使用“进入模块 / 查看入口 / 继续工作流”这类系统导向表达
- 页面中的状态标签应表达用户当前结果或下一步，而不是系统阶段说明
- 删除产品名、系统名、阶段词后，文案仍然要成立

### 1.5 整改方向

后续整改必须围绕下面四个统一目标展开：

- 从“展示很多功能”改成“让用户一眼知道接下来做什么”
- 从“每块都讲一遍自己是什么”改成“页面只保留一次总说明，局部只说动作和数据”
- 从“系统介绍式文案”改成“任务驱动式文案”
- 从“卡片式堆叠”改成“主次分层 + 内容带状流动”

### 1.6 页面结构规范

#### 页面层级模型

后续页面统一按三类模型治理：

- 工作台页
  - 典型页面：`Dashboard / Study Plan / Applications`
  - 标准结构：`一个主任务区 + 一组关键摘要 + 一组次级入口`
- 内容列表页
  - 典型页面：`Question / Knowledge / Community`
  - 标准结构：`一个页头 + 一个筛选/操作区 + 一个主内容区`
- 流程页 / 详情页
  - 典型页面：`Interview / Resume / Application Detail`
  - 标准结构：`一个状态头 + 一个主内容流 + 一个辅助侧栏`

#### 页头规范

- `AppShellHeader` 只保留一套语义来源
- `route meta` 只提供导航语义和最小页面标识，不再默认承担完整说明文案
- 页面渲染时默认只允许 `AppShellHeader` 输出 `title + actions`
- `subtitle` 仅在复杂页面保留，且必须回答“现在可以做什么”
- `kicker` 改为可选，不再默认全站出现

#### 文案规范

- 工作台页允许一段总说明
- 列表页默认不再重复讲“这个模块是干什么的”
- 卡片内部只允许写数据、动作或状态，不再写第二层模块介绍
- 标题、按钮、空状态、表单提示都必须按用户视角重写

#### 卡片规范

- 主卡承载当前主任务或核心数据
- 次卡承载补充信息
- 信息条承载轻量状态，不独立成大卡
- 禁止把每个小信息点都做成独立大卡
- 禁止用多张并列卡代替页面主结构

#### 标签规范

- `hard-chip` 只保留主状态
- `detail-pill` 只保留辅助元信息
- 首屏单区域内标签数量默认限制为 2-3 个，超过必须折叠或降级

#### 动作规范

- 每个页面首屏只能有一个主 CTA
- 其它动作必须降级为次级按钮、文本链接或菜单项
- 禁止两个以上视觉同权的主按钮争抢注意力

### 1.7 全站文案治理规范

#### 文案目标

- 帮用户理解当前状态
- 告诉用户下一步做什么
- 降低理解成本，而不是介绍系统设计

#### 允许的文案类型

- 任务导向
  - 例如：开始今天的训练、上传一份简历开始整理项目问答
- 结果导向
  - 例如：你还有 3 个待处理投递、今天还有 5 个待复习问题
- 操作导向
  - 例如：继续这轮面试、补充这条投递记录

#### 禁止的文案类型

- 系统自我描述
  - 例如：当前已接入真实简历解析
- 阶段化描述
  - 例如：当前已切换为求职训练主线
- 模块说明书
  - 例如：该模块用于承接……
- 研发占位表达
  - 例如：入口已预留、后续将扩展、先搭建入口

#### 改写原则

- 把“系统能力”改写成“用户收益”
- 把“模块定位”改写成“用户当前任务”
- 把“状态标签”改写成“可执行动作或结果”

#### 关键场景要求

- 页标题：必须是用户任务名
- 副标题：必须说明当前可做什么
- Hero 文案：必须说明当前主目标，不得介绍系统背景
- 空状态：必须包含下一步动作
- CTA：必须是清晰动作，不得是泛化入口词
- 表单提示：必须帮助填写，不得重复字段名

#### 文案验收标准

- 不看设计稿，只看首屏文案，也能知道这页该做什么
- 删除系统名后，文案仍成立
- 文案不依赖“阶段、模块、系统”词汇也能表达完整

## 3. 详细差距

### 3.1 产品表达、界面与文案治理

#### 当前代码现状

- 全局 `AppShellHeader` 默认输出 `kicker + title + subtitle`
- `router/index.ts` 为多个页面注入完整 `meta.subtitle`
- 多个页面正文继续补 hero 卡、说明卡、导览卡、摘要卡和章节标题
- `shell-section-card`、`hard-chip`、`detail-pill`、`section-kicker` 在首页、题库、知识库、学习计划、简历、投递、后台中被高频复用
- 文案大量使用系统自述、阶段表述、模块说明和入口说明式表达

#### 为什么还不算完成

- 当前系统不是缺功能，而是页面表达已经影响功能理解
- 用户进入页面后，需要先理解模块，再理解动作，成本过高
- 当标题、说明、卡片和状态标签都在争注意力时，真正的主任务会被淹没
- 当文案以系统视角叙述时，页面无法建立“当前任务 -> 下一步动作”的自然引导

#### 下一轮建议边界

- 统一页头机制
- 统一全站文案口径
- 收紧首屏卡片数量
- 建立页面层级模型与容器职责规则
- 对关键页面逐步做首屏重构和文案重写

#### 最小验收标准

- 核心页面首屏只保留一套主标题语义
- 核心页面首屏只保留一个主 CTA
- 首屏卡片数量和说明层级明显收敛
- 首屏文案全部改为面向用户动作和结果

### 3.2 产品主线与数据看板

#### 当前代码现状

- 路由和导航已经切到“求职训练工作台”心智
- `DashboardPage.vue` 和 `AnalyticsPage.vue` 仍主要消费复盘、面试、卡片或记忆相关指标
- Dashboard 首屏存在 hero、路线卡、重点卡、辅助入口卡并列堆叠
- 页面中仍有阶段表述残留，文案还在解释系统切换了什么，而不是告诉用户现在先做什么
- 投递、简历、学习计划摘要没有充分进入首页或分析页主视图

#### 为什么还不算完成

- 产品叙事已经更新，但核心看板仍未完全反映求职主线
- 首页既在介绍系统主线，又在堆模块入口，缺少唯一主任务
- 文案没有建立“现在做什么 -> 为什么先做这个”的任务节奏

#### 下一轮建议边界

- 首页增加计划执行率、最近简历、投递状态分布、下一节点提醒
- Analytics 加入投递节奏、面试阶段转化、计划完成趋势
- Dashboard 首屏收敛为一个主 hero + 一组关键指标 + 一组动作入口
- 去掉阶段叙述，改成训练动作导向文案

#### 最小验收标准

- Dashboard 至少新增计划、简历、投递三类摘要卡
- Analytics 至少有一块图表或洞察真正来自投递/计划链路
- Dashboard 首屏只保留一个主任务表达
- 文案不再出现阶段性系统描述

### 3.3 题库与知识库

#### 当前代码现状

- 后端题库查询支持 `tag`
- 前端题库页还没有独立标签筛选 UI
- 题库页 `AppShellHeader` 后又叠加筛选卡、详情卡、辅助说明卡
- 题库卡片中标签、说明、洞察过多，信息前置过满
- 知识库已经支持系统资料、用户资料、Word/PDF/TXT/Markdown 上传与检索
- 知识库页头、文档头、上传卡、筛选卡都在解释模块是什么
- 当前没有稳定链路把简历、JD、项目材料沉淀为统一知识库语料
- 文案偏“资料中心介绍”，不够任务导向

#### 为什么还不算完成

- 题库结构已经升级，但用户侧筛选入口还少一块
- 列表页结构不收敛，说明来源重复
- 用户会在“理解这个模块”和“开始筛选/上传/生成卡片”之间来回切换
- 知识库在定位上已经是资料中心，但在数据来源上还没有真正覆盖求职全链路

#### 下一轮建议边界

- 前端补充标签筛选和常用筛选组合
- 为简历、JD、项目文档定义明确的 `business_type` 入库与治理路径
- 列表页只保留一个页头说明源
- 筛选区与说明区合并
- 上传区、空状态、按钮文案全部改成用户动作导向

#### 最小验收标准

- 题库页可按标签筛选
- 至少有一条从简历、JD 或项目资料进入知识库的正式链路
- 题库页与知识库页都只保留一层页头说明
- 上传区和空状态能直接引导用户下一步动作

### 3.4 RAG 与面试联动

#### 当前代码现状

- Chat 已支持学习版、面试版、简短版、项目模式等参数
- 项目模式主要还是 prompt 语气差异，并未稳定注入项目上下文
- Chat 顶部模式切换密度高，没有单一主任务提示
- 面试启动参数中已有 `includeResumeProject`，但没有真正绑定 `resumeId` 或注入 `resume_project`
- Interview 页面配置面板、状态信息、作答区和侧栏说明都偏满
- 文案偏“模式说明”，而不是“现在该怎么提问/作答”

#### 为什么还不算完成

- “项目模式问答”和“结合简历项目面试”在体验层已经出现，但核心数据没有真正打通
- 用户会以为系统在用自己的项目资料，实际多数情况下仍是泛化问答或泛化面试
- 顶部配置和说明密度过高，会削弱当前最重要的问答或作答动作

#### 下一轮建议边界

- Chat 支持显式选择简历或项目上下文
- Interview 支持按简历或项目生成问题与追问
- 模式选择区与说明区合并
- 面试页按“配置前 / 作答中 / 结束后”严格分态
- 文案改成提问与作答动作导向

#### 最小验收标准

- 新增 resume/project 绑定参数并进入服务链路
- 问答与面试结果中能明确显示来自哪份简历或哪个项目
- Chat 顶部只保留当前必要配置
- 面试页不同状态不再同时堆叠说明区

### 3.5 学习计划

#### 当前代码现状

- 已支持生成 7 / 14 / 30 天计划、任务列表、刷新和打卡
- 计划主要根据弱项摘要和模板规则生成
- 页面已经能展示当前计划和任务状态
- Study Plan 页面存在 `AppShellHeader + hero + 参数说明卡 + 计划卡 + 摘要卡 + 辅助信号卡`
- 文案大量解释计划机制，而不是帮助用户开始执行

#### 为什么还不算完成

- 计划具备“能用”的排程能力，但数据驱动程度不够强
- 薄弱点变化趋势和计划执行效果的反馈还不够明显
- 页面既有计划前引导，又有计划后摘要，叙事不收敛
- 用户会被“系统怎么生成计划”吸走注意力，而不是去执行今天的任务

#### 下一轮建议边界

- 引入更多历史训练数据作为生成输入
- 增加任务效果反馈和弱项趋势对比
- 首屏只保留一个计划状态块

### 7.13 跨模块依赖与实施顺序

固定顺序如下：

1. 全局页头、卡片和文案治理先行
2. Dashboard / Study Plan / Resume / Applications 首屏优先整改
3. Question / Knowledge / Chat / Interview 再做结构和联动优化
4. Application / Resume / Chat / Interview 的上下文与复盘链路再做后端串联
5. 最后补 AI 治理、系统配置审计和知识治理增强

依赖关系必须明确：

- 不先统一页头和文案规则，单独改页面会反复返工
- 不先定义知识库语料边界，RAG 和面试联动无法稳定落地
- 不先定义投递事件结构，复盘建议无法形成可计算输入
- 不先收敛工作台与详情页结构，后续数据新增只会继续加重首屏混乱
