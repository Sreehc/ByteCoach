## 审查说明

- 审查日期：2026-05-14
- 审查对象：当前 `frontend/src/pages`、全局布局与路由覆盖到的前端页面
- 审查方式：
  - 已完成本地前端启动与构建验证
  - 公开页与整体壳层做了可运行检查
  - 登录后页面因当前未接通可用登录后数据链路，本次以代码审阅为主
  - 已按 `frontend-design` 复核首屏叙事、视觉中心、信息密度、卡片/标签层级与页面设计意图
  - 已按 `ui-ux-pro-max` 复核可访问性基础项、触控尺寸、反馈状态、响应式路径、导航清晰度与前端性能风险
  - 已按 `ux-writing` 复核标题、按钮、空状态、提示语、系统反馈与导航标签的用户导向表达
- 输出规则：
  - `P0`：直接阻碍用户完成任务，或对用户造成明显误导
  - `P1`：不会立刻阻断，但明显影响理解、效率、判断或转化
  - `P2`：细节体验、表达一致性或视觉收敛问题

## P0

- 页面 / 路由：`/login`、`/register`
  - 现象：登录与注册页左侧主叙事仍在强调“AI 间隔记忆”“知识卡片”，并展示“GitHub 登录暂未开放 / 以后开放”这类未开放能力；登录页还把 `🧠 / 🎯 / 📚` 这类 emoji 当作结构性图标使用。结论来自代码审阅与本地运行核对。
  - 为什么影响用户：这是用户进入产品后的第一屏，会直接把当前产品误导成“记忆卡片工具”，而不是“求职训练工作台”；同时把不可用能力和偏临时感的视觉符号提前暴露，会同时降低信任感与专业感。
  - 建议修改方向：左侧价值区只保留当前正式成立的主线能力，围绕题库、问答、模拟面试、学习计划、简历和投递组织；未开放登录方式不要占首屏；结构性图标统一改成正式图标系统或直接删除宣传卡片。
  - 验收标准：登录和注册首屏删除“AI 间隔记忆”“知识卡片”“暂未开放 / 以后开放”类表述，不再使用 emoji 充当核心功能图标，用户仅通过首屏即可知道这是求职训练平台。

- 页面 / 路由：`/question`
  - 现象：题库页卡片上的“继续追问”“去练表达”直接跳到 `/chat` 和 `/interview`，但没有把当前题目上下文一并带过去；结论来自代码审阅。
  - 为什么影响用户：按钮文案承诺的是“延续当前题目”，实际行为却是“进入一个全新通用页面”，这属于直接的任务断裂和动作失真。
  - 建议修改方向：要么把题目 `id / title / tag / category` 透传到问答和面试页，形成真正的连续动作；要么把按钮文案降级成“去问答页”“去模拟面试页”。
  - 验收标准：从任意题目卡点击后，用户能在目标页直接看到当前题目的预填上下文，或按钮文案不再暗示连续承接。

- 页面 / 路由：`/question`
  - 现象：题库首屏同时包含页头、说明、4 张摘要卡、7 列筛选矩阵和双按钮操作区，信息密度过高；结论来自代码审阅。
  - 为什么影响用户：用户进入题库的主任务是“先看到题，再筛题”，当前首屏把大量筛选和摘要堆在题目前面，会拖慢第一次决策，尤其在中等宽度桌面和移动端更明显。
  - 建议修改方向：收敛为“一行主筛选 + 次级筛选折叠 + 结果列表立即出现”的结构，默认首屏应尽快露出第一批题目。
  - 验收标准：在常见桌面宽度下，进入题库后无需明显下滚即可看到至少一条题目结果；筛选区不再占满整屏。

## P1

- 页面 / 路由：`/knowledge`
  - 现象：知识库首屏同时承载页头、模式切换、长说明、4 个状态统计、文档列表、右侧上传区和完整筛选器；文案大量使用“可使用 / 待处理 / 解析 / 索引”等系统状态词；结论来自代码审阅。
  - 为什么影响用户：用户真正关心的是“我要上传什么、这份资料现在能不能拿来问答”，但当前首屏更像处理管线面板，任务焦点被系统状态吞掉。
  - 建议修改方向：把首屏改成“上传或选择资料 -> 看当前状态 -> 继续问答/生成卡片”的线性结构；系统状态术语仅保留在次级信息层。
  - 验收标准：用户进入知识库后，能在首屏 3 秒内判断“先上传 / 先筛选 / 先打开一份资料”，而不是先理解解析与索引阶段。

- 页面 / 路由：`/knowledge`、`/review`、`/cards`、`/settings`、部分后台页
  - 现象：多处文案仍以系统状态或模块分类为主，例如“可使用”“待处理”“解析”“索引”“账号状态”“数据导出”“系统概览”；结论来自代码审阅。
  - 为什么影响用户：从 `ux-writing` 角度看，这类词对实现方是清楚的，对用户却要先翻译成“我现在能做什么”。文案没有直接回答用户任务，会拖慢理解。
  - 建议修改方向：把系统词改成动作或结果导向表达，例如把“可使用”改成“现在可以提问”，把“待处理”改成“还在准备中”，把“账号状态”改成“账号与邮箱”，把“数据导出”改成“导出我的数据”。
  - 验收标准：页面主标题、状态词和设置标签都优先使用用户能立即采取行动或理解结果的说法，而不是内部处理术语。

- 页面 / 路由：`/interview`
  - 现象：模拟面试开始前配置项过多，方向、岗位、经验、时长、技术范围、上下文、简历、项目、题量、作答模式和最近历史同时堆在同一状态页；结论来自代码审阅。
  - 为什么影响用户：用户在真正开始作答前就需要做过多配置判断，会拖慢启动动作，也削弱“现在先开始一轮面试”的主任务感。
  - 建议修改方向：收敛成“必要配置优先，进阶配置折叠”，并把“最近面试记录”下移或折叠，不和开始配置同屏争注意力。
  - 验收标准：开始前用户只需完成少量必要项即可启动；高级上下文和历史记录不再压过主 CTA。

- 页面 / 路由：`/interview`、`/dashboard` 的面试相关模块
  - 现象：文案频繁使用“诊断”口径，例如“开始诊断”“开始语音诊断”“最近面试诊断结果”“做一次诊断”；同时“开始前设置”“方向”“题量”等标题也偏表单或系统配置视角。结论来自代码审阅。
  - 为什么影响用户：这会把产品体验写成系统评估流程，而不是用户熟悉的“开始一次模拟面试 / 查看上次面试表现 / 继续练表达”。动作词偏抽象，也不够自然。
  - 建议修改方向：把主 CTA 和标题统一到用户任务语义，例如“开始模拟面试”“开始语音面试”“查看上次面试表现”“先选这轮面试怎么练”。
  - 验收标准：面试主线页面不再依赖“诊断”作为核心动作词，用户仅看标题和按钮就知道自己是在开始、继续还是回看一轮面试。

- 页面 / 路由：`/applications`
  - 现象：投递页同时要求用户“记录新岗位”“继续推进当前岗位”“按阶段浏览看板”，首屏主任务不够单一；结论来自代码审阅。
  - 为什么影响用户：录入新岗位和推进旧岗位是两种不同任务，放在同一首屏会让用户难以判断当前优先级。
  - 建议修改方向：根据是否存在在推进岗位决定首屏主任务；若已有当前重点岗位，首屏先强调推进，录入表单降级为次级入口或抽屉。
  - 验收标准：进入投递页后，用户一眼能判断当前是“先推进已有岗位”还是“先录入第一条岗位”。

- 页面 / 路由：`/chat`
  - 现象：问答页顶部仍承载路径选择、资料范围、简历选择、项目选择、会话栏、上下文说明等多层控制；虽然比旧版收敛，但开始提问前的控制密度仍偏高；结论来自代码审阅。
  - 为什么影响用户：用户主任务是“提问”，但顶部过多配置会把认知压力前置，尤其对第一次进入的人不友好。
  - 建议修改方向：默认只保留一条当前提问路径和一个最小上下文入口，其余控制放入“更多设置”。
  - 验收标准：首次进入问答页时，用户无需先读完整个控制区，也能直接开始一次清晰提问。

- 页面 / 路由：`/chat`、`/question`
  - 现象：部分按钮和路径名仍偏功能标签，而不是明确动作，例如“自由提问”“基于资料回答”“简历 / 项目上下文”“继续问答”“去练表达”；结论来自代码审阅。
  - 为什么影响用户：这些词大致能懂，但不够具体，不直接说明点击后会发生什么，尤其在跨页面跳转时更容易产生预期偏差。
  - 建议修改方向：按钮统一改成“动词 + 对象 / 结果”结构，例如“直接提问”“带资料提问”“结合简历提问”“去问这个问题”“去做一轮表达练习”。
  - 验收标准：主要 CTA 和路径切换项都能清楚表达动作结果，减少抽象分类词和省略主语的入口词。

- 页面 / 路由：`/analytics`
  - 现象：分析页首屏同时出现时间范围切换、洞察条、概览卡、趋势图、复习图表和掌握度模块，多块信息使用近似视觉权重；结论来自代码审阅。
  - 为什么影响用户：从 `frontend-design` 的信息层级视角看，用户很难在 3 秒内判断“今天最该看哪条信号”，页面更像把所有分析能力平铺出来，而不是帮助用户先做一次判断。
  - 建议修改方向：首屏只回答一个主问题，例如“当前求职准备是否在推进”；保留 1 个主趋势区和少量关键指标，其余图表下移到第二屏或按主题分组。
  - 验收标准：用户进入分析页后，首屏先看到 1 个明确主结论区，而不是多个同权图表同时争抢注意力。

- 页面 / 路由：`/analytics`、`/review`、`/community` 等含空状态页面
  - 现象：空状态和加载态里高频出现“暂无主线趋势数据”“暂无复习数据”“暂无面试数据”“加载复习数据...”这类结果描述，但缺少更直接的下一步动作；结论来自代码审阅。
  - 为什么影响用户：从 `ux-writing` 的空状态规范看，光说明“没有”还不够，用户更想知道“现在该做什么才能让这里出现内容”。
  - 建议修改方向：空状态统一采用“当前情况 + 下一步动作”结构，例如“这里还没有面试趋势。先完成一轮模拟面试，再回来查看变化。”；加载态也尽量说明在等什么，例如“正在加载今天的复习任务”。
  - 验收标准：核心空状态都包含明确下一步；加载态优先说明正在准备的内容，而不只写“加载中”。

- 页面 / 路由：`/forgot-password`
  - 现象：找回密码页会在 `debugCode` 存在时直接渲染“开发验证码”；结论来自代码审阅。
  - 为什么影响用户：这会把内部调试信息和正式用户流程混在一起，既破坏安全感，也会让环境配置错误直接暴露到终端界面。
  - 建议修改方向：调试验证码仅允许在显式开发环境调试面板展示，正式用户界面只保留发送结果、目标邮箱和有效期。
  - 验收标准：生产可见流程中不再出现任何验证码明文或调试字段。

- 页面 / 路由：全局加载体验
  - 现象：本地构建产物出现明显大包告警，`element-plus`、`echarts`、`ChatPage`、`AdminPage` 和全局 CSS 体积偏大；结论来自本地构建输出。
  - 为什么影响用户：虽然这不是单页布局 bug，但会直接影响首屏加载、路由切换和低性能设备体验，属于用户感知明显的问题。
  - 建议修改方向：拆分后台与图表依赖，收紧全局样式体积，审查 `element-plus` 与图表的按需加载。
  - 验收标准：构建不再出现超大核心包警告；问答页、后台页和图表页的首开成本明显下降。

- 页面 / 路由：`/knowledge`、`/review`、`/cards`、部分后台页
  - 现象：仍存在较多“状态 chip / detail pill / 卡片小模块”高频堆叠的情况，多个轻量状态被做成同权重信息块；结论来自代码审阅。
  - 为什么影响用户：当每块都像重点时，用户反而抓不到真正该看的数据和动作。
  - 建议修改方向：把主状态压缩到 1 到 2 个，把辅助信息收回正文或列表元信息，不再用多个同权标签平铺。
  - 验收标准：同一首屏区域的状态标签数量明显下降，页面主动作与主结果比辅助状态更突出。

## P2

- 页面 / 路由：`/login`、`/register`、`/forgot-password`
  - 现象：左侧品牌价值区和右侧表单区都各自承担了一套标题语义，页面存在“双标题中心”；结论来自代码审阅与本地运行核对。
  - 为什么影响用户：不会直接阻断操作，但会削弱单一主任务，页面更像“双栏宣传 + 表单”而不是“先完成当前动作”。
  - 建议修改方向：保留一个主标题中心，另一侧更多承担辅助说明而非第二个主叙事。
  - 验收标准：每个认证页只保留一个真正的一级叙事中心，另一侧不再和表单标题争主位。

- 页面 / 路由：`/settings`
  - 现象：设置页一级标签命名仍偏“模块名”，例如“账号状态”；结论来自代码审阅。
  - 为什么影响用户：标签名不够动作导向，用户需要先理解信息分类，再决定去哪里操作。
  - 建议修改方向：改成更明确的用户任务口径，例如“账号资料”“安全验证”“登录设备”“登录记录”。
  - 验收标准：设置页标签都能直接回答“我来这里要做什么”。

- 页面 / 路由：`/login`、`/register`、`/settings/account`
  - 现象：账号相关文案仍存在“暂未开放”“以后开放”“其他登录方式”“可使用”这类平台公告式表达；结论来自代码审阅。
  - 为什么影响用户：这类写法更像产品方对功能状态的说明，不像用户此刻可以执行的动作，也会把“当前不能做”的信息放得过重。
  - 建议修改方向：改成更克制的用户口径，例如“GitHub 登录尚不可用”“开放后可在这里启用”，或在没有可行动作时直接不占主位。
  - 验收标准：账号页与认证页不再把未开放能力作为主要说明内容，状态提示优先服务当前能完成的动作。

- 页面 / 路由：`/404`
  - 现象：404 页的主 CTA 是“回到首页”，但未登录用户点击后实际会被守卫带到登录页；结论来自代码审阅。
  - 为什么影响用户：CTA 文案和真实落点不一致，会形成轻微的路径误导。
  - 建议修改方向：未登录态改成“去登录”或“回到首页 / 登录入口”的条件化文案。
  - 验收标准：404 页 CTA 与用户真实去向一致。

- 页面 / 路由：全局布局
  - 现象：固定顶栏 + 页面级 `AppShellHeader` 形成双层头部，虽然语义上已分工，但在高度受限设备上仍然压缩正文可视区域；结论来自代码审阅。
  - 为什么影响用户：不会直接导致错误，但会让列表页和表单页更晚露出真正内容。
  - 建议修改方向：在移动端和中等高度窗口下进一步压缩页头高度，或让部分页面采用更紧凑头部模式。
  - 验收标准：在常见笔记本和移动端视口中，内容首屏可见面积明显提升。

- 页面 / 路由：`/login`、`/register`、`/forgot-password`、`/settings` 及其它表单页
  - 现象：从 `ui-ux-pro-max` 的表单反馈与可访问性基线看，当前代码中已具备基础 `label`、密码显隐和部分 `aria-label`，但未见统一的“提交失败后聚焦首个错误字段 / 错误摘要 / `aria-live` 错误播报 / 跳到主内容”策略；结论来自代码审阅。
  - 为什么影响用户：普通用户仍可完成操作，但键盘用户、屏幕阅读器用户和移动端用户在长表单或标签页结构中更容易丢失当前错误位置，恢复路径不够明确。
  - 建议修改方向：补统一表单可访问性基线，包括首个错误聚焦、错误摘要、重要提示播报与主内容快捷进入。
  - 验收标准：表单提交失败时，焦点会落到第一个错误字段或错误摘要；重要错误和成功提示能被辅助技术正确感知。

## 总结

- 当前前端最大的不是“样式不统一”，而是部分关键页面仍然存在“旧叙事残留、主任务不够单一、动作承诺与真实行为不一致”的问题。
- 结合 `frontend-design` 与 `ui-ux-pro-max` 复核后，问题不只在文案，也在视觉中心不够单一、图表与控制区同权竞争、以及表单与反馈链路的无障碍基线偏弱。
- 结合 `ux-writing` 复核后，文案层面的主要问题也已经明确：系统口吻偏重、按钮动词不够具体、空状态只报结果不指路、设置与状态标签仍在用模块名而不是用户任务名。
- 最应先修的是：
  - 登录 / 注册页的错误产品叙事
  - 题库页的首屏过载
  - 题库卡片到问答 / 面试页的上下文断裂
- 第二批应收敛的是：
  - 知识库、问答、模拟面试、投递页的首屏控制密度
  - 全站状态标签与辅助卡片滥用
  - 首屏加载体积过重带来的性能问题
