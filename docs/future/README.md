# OfferPilot 前端页面体验审查任务书

## 文档定位

本文档用于指导一次面向全站前端页面的体验审查。你现在以产品经理和 UI/UX 设计师的视角检查 OfferPilot 的全部前端页面，不是做代码走查摘要，也不是写路线图。

- 目标是发现会影响理解、决策、操作效率和转化的问题
- 输出以问题清单和修改方向为主，不输出实现日志
- 正式产品能力、接口和设计规范分别以 `docs/prd`、`docs/api`、`docs/design` 为准

## 任务目标

从真实页面体验出发，检查当前前端页面是否存在以下问题：

- 页面显示 bug、重叠、遮挡、溢出、断点错位或滚动异常
- 信息层级混乱、内容冗余、重复标题或重复说明
- 首屏重点不清，用户无法迅速判断当前该做什么
- 文案仍然面向程序员、系统或模块说明，而不是面向用户任务
- 空状态、加载态、错误态没有给出明确下一步动作
- 卡片、标签、按钮、说明区等组件使用失衡，造成视觉噪音
- 导航路径、任务路径或页面内动作路径不清晰
- 任何未列出但会影响用户理解、操作或转化的问题

## 必查维度

### 1. 显示与布局

- 是否存在重叠、遮挡、溢出、错位、滚动冲突
- 响应式断点下是否出现布局断裂、内容被截断、按钮不可点
- 表格、侧栏、弹窗、时间线、图表在不同宽度下是否可读

### 2. 信息架构与重点表达

- 页面首屏是否有唯一主任务
- 是否重复出现标题、副标题、说明卡、hero 描述或引导语
- 是否存在多个视觉同权的主按钮争抢注意力
- 关键信息是否被次要标签、统计卡或装饰信息淹没

### 3. 文案与用户视角

- 页面标题是否表达用户任务，而不是系统结构
- 描述文案是否在告诉用户“现在可以做什么”
- 是否出现程序员口吻、系统口吻、模块说明书口吻
- CTA 是否是明确动作，而不是抽象入口词

### 4. 状态设计

- 空状态是否清楚说明下一步动作
- 加载态是否让用户知道正在等待什么
- 错误态是否能说明问题并提供重试或替代路径
- 成功提示、警告提示是否足够具体

### 5. 组件一致性与视觉噪音

- `AppShellHeader`、卡片、标签、摘要区、说明区的职责是否清晰
- 是否存在卡片、chip、pill 过度使用，导致每块都像重点
- 页面是否通过堆容器而不是清晰结构表达主次关系

### 6. 导航与任务路径

- 用户是否能从首页自然进入下一步动作
- 页面内主要流程是否顺畅，例如筛题、上传资料、开始面试、执行计划、推进投递
- 返回路径、详情页跳转、辅助入口是否清楚

### 7. 可读性与扫描效率

- 首屏是否能在短时间内读懂
- 标题、数据、状态、动作的阅读顺序是否合理
- 表述是否简洁，不需要先理解系统设计才能操作

### 8. 可访问性基础项

- 关键按钮和输入区是否有足够的点击面积
- 文本与背景对比是否足够
- 状态是否只靠颜色表达
- 表单、弹窗、错误提示是否容易理解

### 9. 补充发现

- 任何未在以上分类中列出，但会影响理解、决策、操作或转化的问题，都必须记录

## 审查方法

- 优先以真实页面体验为准，包括实际渲染、交互和响应式表现
- 如果页面无法完整运行，可以结合代码审阅补足判断
- 当结论来自代码审阅而不是页面运行时，必须明确标注
- 不以实现难度替代问题判断，只关注用户是否因此受阻

## 输出格式

输出结果必须先按优先级汇总，再逐条展开。

### 优先级定义

- `P0`：直接阻碍用户完成任务，或造成明显显示错误、强干扰、严重误导
- `P1`：不会立刻阻断，但明显影响理解、效率、判断或转化
- `P2`：细节体验、表达一致性或视觉收敛问题

### 输出顺序

1. 先列 `P0`
2. 再列 `P1`
3. 最后列 `P2`

### 每条问题必须包含

- 页面 / 路由
- 现象
- 为什么影响用户
- 建议修改方向
- 简短验收标准

### 推荐格式

```text
P0
- 页面 / 路由：
- 现象：
- 为什么影响用户：
- 建议修改方向：
- 验收标准：
```

## 审查原则

- 以真实页面体验为准，不以实现难度替代问题判断
- 优先指出会阻碍用户完成任务的问题
- 不把“系统能力介绍”误判为“对用户有价值的内容”
- 页面重点必须服务当前任务，而不是服务模块自我介绍
- 如果无法运行页面，必须明确说明哪些结论来自代码审阅

## 额外要求

- 不要输出“整体还可以”“建议优化 UI”这类空泛结论
- 每条问题都要落到具体页面或具体路径
- 不要把多个独立问题混成一条
- 如果同类问题跨多个页面重复出现，可以先给全局结论，再分别列出受影响页面

## 完成标准

- 审查范围覆盖当前真实前端页面与路由
- 输出至少能区分阻断问题、重要体验问题和细节问题
- 结论能直接交给前端、产品或设计继续整改
- 阅读结果后，团队能直接知道先修什么、为什么修、修到什么程度算完成
