# ByteCoach 项目规划与功能评审

## 1. 项目总体评价

ByteCoach 的定位清晰，目标是通过问答、模拟面试、错题沉淀与学习计划，形成一个围绕 Java 后端面试准备的智能学习闭环。从产品价值看，这个方向具备较强的展示性和可讲述性，适合作为 AI + Java 的项目作品。

当前规划的优点是：

- 用户主链路清晰，覆盖注册登录、学习、面试、复习与计划。
- AI 能力与传统业务能力结合较自然，不是纯聊天壳项目。
- 模块拆分已经具备平台雏形，后续可延展到 RAG、画像、长期记忆等。

当前规划的主要问题是：

- 功能范围偏大，远超 1 人 2-4 周的稳定交付范围。
- RAG、评分、追问、画像、计划都想在一期做全，风险较高。
- 某些模块边界重叠，比如普通聊天和知识问答、题库和知识库、画像和统计。
- 如果不先收敛，一期很容易变成“页面很多但闭环不稳”的项目。

## 2. 功能完整性评分

- 当前功能完整性评分：`7.5 / 10`

评分依据：

- 已覆盖核心角色、主要业务对象和关键业务链路。
- 具备用户、内容、问答、面试、错题、计划等一条可闭环的学习路径。
- 但在知识片段、面试会话、任务明细、画像落地等方面还存在关键缺口。

## 3. 功能合理性评分

- 当前功能合理性评分：`6.5 / 10`

评分依据：

- 必要功能基本都在，但部分功能过早规划到一期。
- AI 相关能力没有明确分层，一期容易把工程复杂度拉高。
- 用户价值高的功能和演示价值高的功能混在一起，缺乏优先级收敛。

## 4. 核心业务闭环分析

建议将 ByteCoach 一期闭环定义为：

`注册/登录 -> 首页引导 -> Java 问答 -> 模拟面试 -> AI 评分 -> 自动沉淀错题 -> 生成学习计划 -> 每日复习 -> 首页查看进度与薄弱点`

### 4.1 当前闭环是否成立

整体上已经具备闭环雏形，但要真正落地，需要补齐以下支撑点：

- 知识库闭环缺口：
  - 仅有知识文档表不够，缺少知识片段层，RAG 无法真实支撑引用和检索。
- 面试闭环缺口：
  - 仅有答题记录不够，缺少一场面试的会话实体，无法描述多轮面试过程。
- 学习计划闭环缺口：
  - 仅有计划主表不够，缺少任务明细，无法支持每日任务和完成状态。
- 用户画像闭环缺口：
  - 一期不应做复杂画像系统，但必须有可计算的统计结果用于首页和计划。

### 4.2 用户主流程是否完整

从用户视角看，一期最重要的主流程是完整的：

1. 注册登录并进入系统。
2. 在首页看到当前学习状态和推荐入口。
3. 通过问答或面试进入学习行为。
4. 得到 AI 反馈、标准答案和评分。
5. 低分内容自动进入错题本。
6. 根据错题与薄弱项生成学习计划。
7. 在后续访问中持续复习并看到进展。

这条路径能够支撑产品演示、项目答辩和后续研发。

## 5. 各模块详细评审

### 5.1 用户模块

- 模块作用：
  - 提供身份体系、数据隔离、学习记录绑定和权限控制。
- 当前功能是否必要：
  - 必须做。
- 是否存在重复：
  - 无明显重复。
- 是否存在过度设计：
  - 当前设计较克制。
- 是否需要补充：
  - 建议补充 `last_login_time`、`source`、`remark` 字段。
  - 一期采用 JWT 鉴权，支持基础登录状态管理即可。

### 5.2 Agent 问答模块

- 模块作用：
  - 作为产品核心入口，承担学习助手与知识检索体验。
- 当前功能是否必要：
  - 必须做。
- 是否存在重复：
  - “普通聊天”和“知识问答”容易让用户产生心智混乱。
- 是否过度设计：
  - 如果一期同时加入 Tool Calling、多模型、多 Agent，会过度。
- 调整建议：
  - 合并为统一 Chat 模块。
  - 支持两种模式：普通问答、知识库问答。
  - 一期必须支持：会话列表、消息历史、Markdown 展示、引用片段展示。

### 5.3 知识库模块

- 模块作用：
  - 为 Java 面试问答提供结构化语料，是 RAG 的基础。
- 当前功能是否必要：
  - 必须做，但要收缩范围。
- 是否存在重复：
  - 与题库模块在“内容管理”层会有边界接近，但对象不同。
- 是否过度设计：
  - 用户上传、在线解析、切分、向量化全链路不适合一期。
- 调整建议：
  - 一期采用“内置资料库 + 管理端导入”。
  - 用户端只提供列表、分类查看、检索测试。
  - 文档切分和向量化先作为后台处理能力和扩展点。

### 5.4 面试模块

- 模块作用：
  - 体现产品差异化，是第二核心亮点。
- 当前功能是否必要：
  - 必须做。
- 是否存在重复：
  - 与题库模块共享题目来源，但业务场景不同。
- 是否过度设计：
  - 同期做多轮追问、复杂自适应评分策略会过重。
- 调整建议：
  - 一期支持按方向开始一场面试。
  - 每场 3-5 题。
  - 每题提交答案后返回分数、点评、标准答案和单次追问。
  - 低分自动入错题本。

### 5.5 题库模块

- 模块作用：
  - 提供统一题源，支撑面试、错题、计划和内容运营。
- 当前功能是否必要：
  - 必须做。
- 是否存在重复：
  - 与知识库不是重复关系，一个偏题目，一个偏资料。
- 是否过度设计：
  - 标签运营、复杂统计可后置。
- 调整建议：
  - 一期保留 CRUD、分类筛选、难度筛选、标准答案管理。
  - 频率字段可作为静态展示或后台手动维护。

### 5.6 错题本模块

- 模块作用：
  - 承接面试和问答反馈，形成复习入口。
- 当前功能是否必要：
  - 必须做。
- 是否存在重复：
  - 与学习计划存在协同，不重复。
- 是否过度设计：
  - 手动新增错题不是一期重点。
- 调整建议：
  - 核心是低分自动沉淀。
  - 提供列表、详情、掌握状态、再次练习入口。
  - 错误原因以 AI 点评和标准答案对比为主。

### 5.7 学习计划模块

- 模块作用：
  - 将薄弱点和错题转化为可执行学习动作。
- 当前功能是否必要：
  - 必须做，但必须轻量。
- 是否过度设计：
  - 动态调整、个性化推荐增强不适合一期。
- 调整建议：
  - 一期只做计划生成、任务拆分、完成状态更新。
  - 计划输入应包含目标方向、学习周期、每日时长。

### 5.8 用户学习画像模块

- 模块作用：
  - 为首页、计划、推荐提供统计支撑。
- 当前功能是否必要：
  - 需要结果，不需要一期独立成大模块。
- 是否过度设计：
  - 是，目前风险最高。
- 调整建议：
  - 一期不单独做复杂画像中心。
  - 以答题记录、计划完成率、错题分类聚合出统计视图即可。

### 5.9 系统管理模块

- 模块作用：
  - 支撑内容管理、用户管理和系统运营。
- 当前功能是否必要：
  - 需要，但只做最小后台。
- 是否过度设计：
  - 权限中心、审计日志、配置中心都可后置。
- 调整建议：
  - 一期只保留用户管理、题库管理、文档管理、分类管理。

## 6. 缺失功能清单

以下内容不是全部都要一期上线，但从设计完整性看是当前文档中缺失的关键点：

- 知识片段管理
- 面试会话实体
- 学习计划任务实体
- 首页聚合统计接口
- 错题再次练习入口
- 问答引用片段返回结构
- 新用户首次使用引导
- 学习方向选择机制
- 计划生成输入参数定义

## 7. 建议后置的功能清单

以下功能建议后置到二期或三期：

- 用户上传资料并实时处理
- 多轮复杂追问状态机
- 多 Agent 编排
- 知识图谱
- 用户长期记忆
- Java 代码运行沙箱
- RAG 效果评估平台
- AI 评分效果评估平台
- Prompt 可视化配置中心
- 多模型路由与供应商切换

## 8. 项目经理结论

ByteCoach 适合做成一个“AI 学习闭环 + Java 面试场景”的作品项目，但一期必须收缩功能范围，优先保证主闭环能跑通、关键页面有演示效果、AI 结果可解释、系统结构清晰可扩展。

如果按文档建议推进，一期可以形成一个足够完整、足够聚焦、适合答辩和展示的 MVP。后续再逐步增强 RAG 效果、面试深度和画像能力，会比一开始追求“大而全”更稳。
