# ByteCoach 开发计划与交付说明

## 1. 项目目标

ByteCoach 一期面向求职作品集和项目答辩场景，目标是在较短周期内交付一个可运行、可展示、可扩展的 Java 面试智能学习 Agent 系统。

## 2. 角色视角结论

### 2.1 项目经理视角

- 必须控制范围，避免一期功能爆炸。
- 应优先保障业务闭环和核心亮点，而不是追求全部高级能力。
- 交付物必须可验证、可展示、可讲解。

### 2.2 产品经理视角

- 产品核心价值不是“聊天”，而是“学习闭环”。
- 主页、面试、错题和计划是最能体现产品价值的功能链。
- 首次引导、快捷入口和过程反馈决定用户是否能感知系统价值。

### 2.3 技术架构师视角

- 一期应采用单体后端 + 前后端分离结构。
- AI 能力必须统一后端封装。
- RAG 与面试评分必须保留扩展位，但不能过度设计。

## 3. 推荐开发排期

假设资源为 `1 人 2-4 周`，建议按以下顺序推进。

### 第 1 周

- 完成需求收敛与文档基线
- 搭建前后端工程骨架
- 完成用户模块与鉴权
- 完成数据库基础表设计
- 完成统一返回体、异常、日志、跨域和 AI 网关骨架

### 第 2 周

- 完成题库、分类、知识库基础能力
- 完成 Dashboard 聚合接口
- 完成 Chat 问答主流程
- 完成基础 AI 封装与 RAG 扩展位

### 第 3 周

- 完成模拟面试主流程
- 完成错题自动沉淀
- 完成学习计划生成与任务管理

### 第 4 周

- 完成 `/admin` 路由下的后台管理最小闭环
- 联调、补充 mock/演示数据
- 完成 README、答辩材料和演示脚本

## 4. 交付物清单

一期建议交付以下内容：

- 前端项目代码
- 后端项目代码
- 数据库 SQL 初始化脚本
- 项目 README
- 接口文档
- 系统设计文档
- 产品规划与迭代文档
- 演示账号与演示脚本

## 5. 项目答辩表达建议

### 5.1 项目亮点建议

- 不是普通聊天项目，而是围绕 Java 面试学习闭环设计。
- 不是简单接大模型，而是结合题库、知识库、面试、错题和计划形成产品链路。
- 一期采用可扩展的 RAG 和 AI 架构，为二期增强留出空间。

### 5.2 演示顺序建议

1. 登录进入 Dashboard
2. 展示当前学习状态
3. 发起一次知识问答
4. 开始一场模拟面试
5. 展示评分、点评和追问
6. 展示错题自动沉淀
7. 展示学习计划生成结果
8. 展示后台题库和文档管理

### 5.3 答辩时应强调的设计取舍

- 为什么一期不做用户上传和复杂 Agent
- 为什么先做内置知识库
- 为什么把画像做成派生统计而不是复杂推荐系统
- 为什么选择单体架构而不是微服务

## 6. 主要风险与应对方案

### 6.1 范围过大

- 风险：
  - 需求过多导致延期或质量下降。
- 应对：
  - 严格按 MVP 清单推进，后置复杂功能。

### 6.2 RAG 效果不好

- 风险：
  - 知识库回答不稳定、引用价值弱。
- 应对：
  - 一期先采用人工整理的高质量 Java 资料。
  - 控制检索链路，先保证结果可解释。

### 6.3 AI 回答不准确

- 风险：
  - 影响用户信任和项目可信度。
- 应对：
  - 展示引用来源。
  - 对面试评分与标准答案做结构化输出。

### 6.4 前后端工作量过大

- 风险：
  - 页面和接口过多导致联调时间不足。
- 应对：
  - 先完成五个高价值页面：登录、Dashboard、Chat、Interview、Wrong/Plan。
  - 后台只做最小管理能力。

### 6.5 API 成本问题

- 风险：
  - 多轮问答和评分成本高。
- 应对：
  - 控制题数和追问次数。
  - 统一模型供应商，简化链路。

### 6.6 部署与运维复杂度

- 风险：
  - 多组件部署导致环境搭建难。
- 应对：
  - 一期优先单体部署。
  - 提供 Docker Compose 或本地启动说明。

## 7. 最终结论

ByteCoach 具备成为一个高质量求职作品项目的潜力，前提是一期不追求“大而全”，而是把“用户学习闭环 + AI 核心亮点 + 工程化骨架”三件事做好。
