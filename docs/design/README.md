# UI / UX 设计

## 文档说明

本文档描述 OfferPilot 当前前端界面的正式设计规则、页面层级和交互约束。重点不是展示模块有多少，而是让用户进入页面后快速判断“现在做什么”。

- 本文档只记录当前成立的设计规范
- 前端页面体验审查任务统一见 [docs/future/README.md](../future/README.md)
- 已过时的阶段路线、差距清单和实施日志不在本文件保留

## 设计目标

- 让用户进入首页后立刻理解求职训练主线
- 把题库、知识库、问答、面试、计划、简历、投递组织成连续工作流
- 让每个页面都能在首屏明确主任务，而不是堆叠说明和入口
- 保持前台训练页、账户页和后台治理页在节奏与层级上的一致性

## 视觉系统

### 设计风格

OfferPilot 采用简洁、专业、工作台式的产品风格。页面优先服务“找到下一步动作”，不把大量说明卡、状态标签和视觉装饰放在首屏争抢注意力。

### 配色语义

基于 CSS 变量的语义主题系统，支持亮色/暗色模式。

| 用途 | Token / class | 说明 |
|------|---------------|------|
| 页面背景 | `--page-bg` | 页面外层背景 |
| 主面板 | `--panel-bg` | 页面核心面板 |
| 次面板 | `--panel-muted` | 提示、空状态、弱强调区域 |
| 交互底色 | `--interactive-bg` | 输入区、分段按钮、次按钮 |
| 主文本 | `--text-primary` | 标题与正文 |
| 次文本 | `--text-secondary` | 描述和辅助信息 |
| 弱边框 | `--border-subtle` | 卡片和列表边框 |
| 主强调色 | `--accent-amber` | 核心 CTA |
| 辅助强调色 | `--accent-cyan` / `--accent-coral` / `--accent-lime` | 信息、警示、完成态 |

页面和共享组件优先使用语义 token 或共享 class，不在页面内重复写死局部颜色。

### 字体

- 正文：`Noto Sans SC`
- 标题：`Barlow Condensed`
- 数字/代码：`JetBrains Mono`

### 圆角

- 小控件：`--radius-sm`
- 卡片：`--radius-md`
- 大面板：`--radius-lg`

## 信息架构

### 一级导航

桌面端和移动端都围绕以下优先级组织：

| 分组 | 页面 |
|------|------|
| 主线训练 | 首页、题库、知识库、问答、模拟面试、学习计划、简历助手、投递管理 |
| 洞察 | 数据分析 |
| 辅助强化 | 卡片强化、复习中心 |
| 外部协作 | 社区 |
| 治理 | 管理后台 |
| 个人中心 | 设置 |

### 首页工作台

首页是求职训练工作台，不是功能总览页。首页首屏默认承担三件事：

- 告诉用户当前最应该先做什么
- 展示少量关键摘要，帮助判断当前状态
- 给出一条主 CTA 和少量次级入口

### 移动端入口

- 底部导航保留高频主线入口
- 其余模块通过“更多”或侧边抽屉进入
- 移动端优先保证问答、面试、题库、知识库和工作台访问效率

## 页面层级模型

后续页面统一按三类模型设计：

### 工作台页

- 典型页面：`/dashboard`、`/study-plan`、`/applications`
- 推荐结构：一个主任务区 + 一组关键摘要 + 一组次级入口
- 首屏重点是“先做什么”，不是解释模块边界

### 内容列表页

- 典型页面：`/question`、`/knowledge`、`/community`
- 推荐结构：一个页头 + 一个筛选/操作区 + 一个主内容区
- 列表页默认不重复写第二层模块介绍

### 流程页 / 详情页

- 典型页面：`/interview`、`/resume`、`/applications/:id`
- 推荐结构：一个状态头 + 一个主内容流 + 一个辅助侧栏
- 不同流程状态应分态展示，避免配置、执行、结果同屏堆叠

## 页面结构规范

### 主布局

```text
┌──────────────────────────────────────────────────┐
│ AppShellHeader                                   │
│ Logo / 标题 / 搜索 / 通知 / 头像                 │
├──────────┬───────────────────────────────────────┤
│ NavRail  │ 页面内容区                            │
│ 主线导航 │ 工作台 / 列表 / 表单 / 详情            │
├──────────┴───────────────────────────────────────┤
│ MobileNavBar（移动端）                           │
└──────────────────────────────────────────────────┘
```

- 桌面端：左侧固定导航，右侧内容区滚动
- 移动端：内容优先，导航收敛到底部和抽屉

### 页头唯一语义来源

- `AppShellHeader` 只承载页面标题和动作区
- 路由 `meta.title` 负责最小页面标识
- 页面正文不再重复追加另一套完整 hero 标题语义
- `subtitle` 只在确有必要时显式传入，必须回答“你现在可以做什么”
- `kicker` 不是默认配置，只在真正需要章节定位时使用

### 首屏唯一主任务

- 一个页面首屏只能有一个一级标题
- 一个页面首屏只能有一个主 CTA
- 非告警页面首屏不应同时堆叠过多状态标签
- 首屏默认不超过 3 个核心面板
- 相同含义的标题、说明、摘要和入口不应重复出现

### 推荐骨架

```html
<div class="space-y-6">
  <AppShellHeader>
    <template #actions>
      <!-- 主 CTA 与次级动作 -->
    </template>
  </AppShellHeader>

  <section class="shell-section-card p-6">
    <!-- 主任务区或筛选区 -->
  </section>

  <section class="shell-section-card p-6">
    <!-- 主内容区 -->
  </section>
</div>
```

## 文案与内容规范

### 文案目标

- 帮用户理解当前状态
- 直接告诉用户下一步做什么
- 降低理解成本，而不是介绍系统设计

### 页面文案红线

- 所有页面文案必须面向用户，不面向系统
- 禁止把“模块定位、研发进度、系统能力说明”写成页面主文案
- 页面标题必须表达用户任务，不表达系统结构
- 副标题必须回答“你现在可以做什么”
- 空状态必须包含下一步动作
- CTA 必须是用户动作，而不是抽象入口词

### 允许的文案类型

- 任务导向：开始今天的训练、上传简历、记录投递、继续这轮面试
- 结果导向：还有 3 条待推进投递、今天还有 5 个待复习问题
- 操作导向：继续追问项目、补充这条岗位记录、查看这轮面试反馈

### 禁止的文案类型

- 系统自我描述：当前已接入、当前预留、已切换为、后续将扩展
- 模块说明书：该模块用于承接、该功能负责管理
- 研发占位表达：先搭建入口、暂时保留、等待后续能力

### 文案验收标准

- 不看设计稿，只看首屏文案，也能知道这页该做什么
- 删除产品名、系统名后，文案仍然成立
- 不依赖“模块、系统、阶段”这些词，也能表达完整动作

## 组件与容器规范

### 核心容器

- `shell-section-card`：主内容面板，不作为所有小信息块的默认容器
- `surface-panel` / `surface-panel-muted`：普通信息和弱强调信息
- `interactive-panel`：表单、筛选器、输入区

### 卡片分层

- 主卡：承载当前主任务或核心数据
- 次卡：承载补充信息
- 信息条：承载轻量状态，不单独做成大卡

禁止把每个小信息点都做成独立大卡，或用多张并列卡代替页面主结构。

### 标签规范

- `hard-chip` 只保留主状态
- `detail-pill` 只保留辅助元信息
- `section-kicker` 只用于真正的章节分割
- 首屏单区域标签默认控制在 2 到 3 个，超出应折叠或降级

### 按钮优先级

- 主 CTA 使用 amber 强调色
- 次级操作使用 neutral / outline 风格
- 危险操作使用显式危险态或二次确认
- 同一屏内不应出现两个以上视觉同权的主按钮

## 交互规则

### 列表与表格

- 表格统一使用 Element Plus `el-table`
- 列表页优先提供搜索和主筛选条件
- 分页统一使用 `pageNum / pageSize`

### 弹窗与确认

- 编辑使用 `el-dialog`
- 删除、封禁、撤销等危险操作必须二次确认

### 加载、空状态与错误态

- 表格使用 `v-loading`
- 页面级使用 skeleton 或占位卡片
- 空状态必须告诉用户下一步做什么
- 错误态必须说明当前无法完成的动作和可重试路径

### 提示信息

- 成功：`ElMessage.success`
- 错误：`ElMessage.error`
- 警告：`ElMessage.warning`
