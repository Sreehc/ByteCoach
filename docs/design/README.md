# UI/UX 设计

## 设计系统

### 设计风格

ByteCoach 采用简洁、专业、工作台式的设计风格，目标是在学习场景下减少视觉干扰，让用户快速回到主任务：上传资料、生成卡片、复习、问答和面试诊断。

### 配色方案

基于 CSS 变量的语义主题系统，支持亮色/暗色模式。深色模式固定使用炭灰 / 墨褐表面体系，不再使用深蓝作为页面底色、面板主底色、抽屉主底色或图表默认主色。

| 用途       | 语义 token                                           | 说明                                 |
| ---------- | ---------------------------------------------------- | ------------------------------------ |
| 页面背景   | `--page-bg`                                          | 页面和全局外层背景                   |
| 主面板     | `--panel-bg`                                         | 页面主要内容面板、弹层、抽屉         |
| 次面板     | `--panel-muted`                                      | 空状态、提示区、弱强调区域           |
| 交互底色   | `--interactive-bg`                                   | chip、分段按钮、次级按钮、输入辅助区 |
| 交互 hover | `--interactive-hover`                                | hover / active 背景                  |
| 主文本     | `--text-primary`                                     | 标题、正文主内容                     |
| 次文本     | `--text-secondary`                                   | 描述、说明、元信息                   |
| 三级文本   | `--text-tertiary`                                    | 弱提示、计数、辅助标签               |
| 弱边框     | `--border-subtle`                                    | 页面面板、列表、表单边框             |
| 强调色     | `--accent-amber`                                     | 主操作、关键路径                     |
| 辅助色     | `--accent-cyan` / `--accent-coral` / `--accent-lime` | 信息、风险、完成状态                 |
| 骨架屏     | `--skeleton-bg` / `--skeleton-shimmer`               | SkeletonBlock 和页面 loading         |

页面和共享组件优先使用语义 token 或共享 class，例如 `text-secondary`、`surface-panel`、`surface-panel-muted`、`interactive-panel`、`skeleton-line`。避免在页面内写死 `slate-*`、`dark:bg-slate-*`、`rgba(255,255,255,...)`、`#365ab0`、`#07111f` 等局部颜色。

### 字体

- 正文：Noto Sans SC
- 标题：Barlow Condensed
- 数字/代码：JetBrains Mono, monospace

### 圆角

- 小组件：`--radius-sm`
- 卡片：`--radius-md`
- 大面板：`--radius-lg`

## 组件规范

### shell-section-card / surface panel

页面主要面板统一使用 `shell-section-card` 或语义 surface class：

```css
.shell-section-card {
  background: var(--panel-bg);
  border-radius: var(--radius-md);
  border: 1px solid var(--border-subtle);
}
```

### metric-card

指标卡片，用于 Dashboard 和数据展示：

```html
<article class="metric-card">
  <p class="metric-label">指标名称</p>
  <p class="metric-value">123</p>
</article>
```

### section-kicker

页面/区块的标签文字，小号大写强调色：

```css
.section-kicker {
  font-size: 11px;
  font-weight: 600;
  letter-spacing: 0.24em;
  text-transform: uppercase;
  color: var(--bc-accent);
}
```

### action-button

主操作按钮，使用 amber 语义强调色：

```css
.action-button {
  background: linear-gradient(180deg, #c4791e 0%, var(--bc-amber) 100%);
  color: white;
  border-radius: var(--radius-md);
  font-weight: 600;
}
```

### accent-link

强调链接，使用主强调色：

```css
.accent-link {
  color: var(--bc-accent);
  font-weight: 600;
  text-decoration: underline;
}
```

- 使用 Element Plus 的 `el-table`
- 斑马纹（stripe）

- 编辑操作使用 `el-dialog`

- 使用 Element Plus 的 `el-pagination`

### 加载状态

- 表格：`v-loading` 指令

- 页面级：SkeletonBlock 组件

### 空状态

- 使用 EmptyState 组件
- 图标 + 标题 + 描述文字

- 成功：`ElMessage.success()`
- 错误：`ElMessage.error()`
- 警告：`ElMessage.warning()`

#### 核心原则

- 核心转化路径优先：先保证 `知识库 -> 生成卡片 -> 今日卡片 -> 复习` 链路清晰，再处理辅助页面装饰和解释。

- 页面文案服务行动：标题、副标题、说明、空状态和按钮文案不重复解释同一件事。
- 工作台页面保持操作密度：Dashboard、Cards、Review、Analytics、Chat、Interview 都应更像学习工具，而不是说明页或营销页。

### 1. 总体判断

ByteCoach 当前前端已经有一条相对清晰的学习主线：`知识库 -> 今日卡片 -> 复习 -> 问答 / 面试 -> 数据分析`。其中 `Dashboard`、`Cards`、`Review` 三个核心页面的产品方向基本一致，说明产品主任务不是散的。

当前的主要问题不在“缺页面”，而在“三层表达同时堆上来”：

- 产品层：很多页面同时在讲目标、讲路径、讲说明，导致重点不够尖锐。
- UI 层：全局风格大体统一，但学习页、设置页、后台页、表格页、图表页之间的密度和气质不稳定。
- 文案层：标题、副标题、说明文案、标签文案普遍偏多，且经常重复同一件事。

一句话结论：

- 产品主线是对的，但首屏焦点还不够狠。
- UI 有统一基础，但信息层级和模块节奏不够稳定。
- 文案冗余是当前最普遍的问题，已经开始直接影响布局和阅读效率。

### 2. 全局问题

#### P0 标题、副标题、说明文案重复，首屏信息密度失真

- 归类：`复合问题`
- 影响：
  很多页面同时出现页面标题、副标题、分区标题、说明文案和提示性卡片，重复传达同一件事。结果不是更清楚，而是让用户更慢找到真正该点的按钮和真正该看的数据。移动端会更明显，因为首屏高度更紧。
- 典型位置：
  `router/index.ts` 的大部分 `subtitle`、`AppShellHeader.vue`、`DashboardGuideCard.vue`、`AnalyticsPage.vue`、`CommunityQuestionDetail.vue`、`SettingsPage.vue` 下各个 tab 子页。
- 建议改法：
  默认只保留一层页面级表达：
  `标题` 或 `标题 + 必要说明`。
  如果标题已经说清楚页面用途，就删除副标题。
  分区内也遵循同一规则：如果分区标题已经成立，就不要再补一段解释性说明。

#### P0 设置页存在“父标题 + 子标题 + 子页标题”三级堆叠

- 归类：`复合问题`
- 影响：
  `SettingsPage` 顶部已经有“账户设置”，tab 内的 `DeviceManagePage`、`LoginHistoryTab`、`TwoFactorTab`、`DataExportTab` 又各自再放一套 `AppShellHeader compact`。真实界面里会形成一页两个标题区，造成纵向空间浪费，也让用户感觉自己被迫“重新进入一个页面”。
- 典型位置：
  `SettingsPage.vue`、`DeviceManagePage.vue`、`LoginHistoryTab.vue`、`TwoFactorTab.vue`、`DataExportTab.vue`
- 建议改法：
  设置页保留一层总标题即可。
  tab 内容区只保留必要的小标题或工具栏，不再重复使用 `AppShellHeader`。

#### P1 学习页主线正确，但导航权重还不够明确

- 归类：`产品问题`
- 影响：
  桌面端左侧 `NavRail` 给了 8 到 9 个等权入口，视觉上都是同级；移动端底部栏只保留 5 个入口，但缺少一个“更多”或“次级模块”的明确出口心智。产品其实有主链路，但导航没有完全把“主任务”与“辅助任务”拉开。
- 典型位置：
  `NavRail.vue`、`MobileNavBar.vue`
- 建议改法：
  左侧导航按层级分组，不要全部等权：
  `主任务：今日卡片 / 复习 / 知识库`
  `辅助：问答 / 面试 / 数据`
  `外部：社区`
  `管理：后台`
  移动端保留现有 5 个主入口，但需要一个更明确的“更多”入口，或者让汉堡菜单承担清晰的次导航角色，而不是隐藏补丁。

#### P1 学习页与后台页视觉系统没有完全收口

- 归类：`UI问题`
- 影响：
  学习主链路页大量使用 `cockpit-panel`、玻璃化背景、柔和色块和强调色芯片；后台和设置页则更接近 Element Plus 默认表格/表单页。看起来不像同一套系统内的不同角色页面，更像两套半融合界面。
- 典型位置：
  `AnalyticsPage.vue`、`DashboardPage.vue` 对比 `AdminOverviewTab.vue`、`AdminUserTab.vue`、`AdminContentReviewTab.vue`
- 建议改法：
  不需要把后台做成和学习页一样“华丽”，但至少统一：
  卡片圆角、按钮主次、筛选区高度、表格外层容器、图表底色、状态色语义。
  后台建议走更简洁、更工具化的同品牌分支，而不是默认组件风格裸露。

#### P1 图表风格与页面底色冲突，数据可信感被削弱

- 归类：`UI问题`
- 影响：
  `Analytics` 和 `AdminOverview` 中的图表区域明显比页面更暗、更重，尤其在浅色大背景上会像单独贴了一块深色屏幕，和其他模块风格断开。图表本应帮助理解，而不是抢视觉重心。
- 典型位置：
  `AnalyticsPage.vue`、`AdminOverviewTab.vue`
- 建议改法：
  统一图表容器底色和站内 panel 体系。
  学习分析页建议弱化容器存在感，突出数据线和读值关系；后台图表建议补充标题、时间范围和图例的结构层级，不要只剩一块图。

#### P1 文案术语不统一，降低学习型产品的专业感

- 归类：`文案问题`
- 影响：
  同一类动作和对象在多个页面有不同叫法，例如：
  `今日卡片 / 记忆任务 / 今日任务`
  `面试诊断 / 诊断 / 做一次诊断`
  `学习社区 / 社区`
  `加入今日卡片 / 生成复习卡片 / 补生成卡片`
  对学习工具来说，这会削弱系统感和可预期性。
- 建议改法：
  建一份最小术语表，并全站统一：
  `页面名、对象名、状态词、按钮词` 各只保留一套主表达。

#### P1 芯片、说明、计数过多，页面经常“每块都像重点”

- 归类：`UI问题`
- 影响：
  `detail-pill`、`hard-chip`、状态条、说明文案、数量标签被大量使用，导致真正的重要数据和 CTA 反而不够突出。尤其在 `Dashboard`、`Review`、`InterviewDetail`、`Leaderboard`、`Cards` 中比较明显。
- 建议改法：
  控制单屏“强调样式”数量。
  一块区域里最多保留一个主强调：
  `主 CTA` 或 `主数据` 或 `主状态`，不要三者同时高亮。

#### P2 桌面快捷键提示直接出现在导航底部，对移动心智没有价值

- 归类：`产品+UI问题`
- 影响：
  侧边栏底部的快捷键卡片在桌面端有帮助，但其表达方式像一个固定功能模块。对触屏场景没有价值，对普通用户也不是高频优先信息。
- 典型位置：
  `NavRail.vue`
- 建议改法：
  桌面端可保留，但弱化存在感。
  移动端不要复用这类提示。
  搜索和收起侧栏的快捷键更适合放入帮助浮层或首次引导，而不是常驻卡片。

### 3. 分页面问题

#### Dashboard

##### P0 空数据时首屏信息重复过多

- 归类：`复合问题`
- 影响：
  空数据状态下同时出现：
  首页标题、副标题、首次使用引导卡、主任务 Hero、指标卡、快捷入口、进阶分析折叠。
  这些模块都在告诉用户“先去生成卡片”，造成重复教育。
- 建议改法：
  空数据时保留一个强主模块即可：
  `先生成第一组卡片 + 主按钮`
  其余入口弱化或折叠。

##### P1 快捷入口卡文案层级过满

- 归类：`文案+UI问题`
- 影响：
  每张快捷卡同时有 `label + badge + title + desc`，信息层级太多，且很多内容是重复的。
- 建议改法：
  缩成两层：
  `标题 + 一行说明`
  badge 只留给确实需要表达状态的卡片。

##### P1 “展开进阶分析”逻辑合理，但默认折叠文案仍在抢视线

- 归类：`产品问题`
- 影响：
  主任务已经成立，但进阶分析入口还是以完整按钮+完整标题形式存在，仍在拉扯首屏注意力。
- 建议改法：
  入口更轻一点，像“查看更多分析”，不要和主任务按钮一个重量级。

#### Cards

##### P0 无 deck 时页面空壳过多，主动作不够集中

- 归类：`复合问题`
- 影响：
  无 deck 时，页面同时展示标题区、空 Hero、四个统计框、deck 切换区、进度区、底部空面板。用户会看到很多“空结构”，但真正的可执行动作只有去知识库。
- 建议改法：
  无 deck 状态收敛为一个主空态：
  `还没有卡片 -> 去知识库生成第一组卡片`
  其余统计和切换模块直接隐藏。

##### P1 中英文混用降低产品完成度

- 归类：`文案问题`
- 影响：
  页面大量使用 `deck`，包括 `0 组 deck`、`切换当前 deck`、`设为当前 deck`。同页其余内容是中文，风格不统一。
- 建议改法：
  统一改为 `卡组` 或 `卡片组`。

##### P1 空状态按钮“今日已完成”会造成误读

- 归类：`产品问题`
- 影响：
  在没有卡片可学的情况下出现一个禁用的“今日已完成”，容易让人误以为系统已经帮他完成，而不是还没开始。
- 建议改法：
  改成 `还没有今日任务` 或直接去掉，保留唯一主动作。

#### Knowledge

##### P0 用户页出现“检索测试”，产品边界不干净

- 归类：`产品问题`
- 影响：
  `检索测试` 更像内部调试或高级运维能力，不适合直接暴露在普通用户知识库页面。它会把“知识资料管理”与“系统检索验证”混在一起。
- 建议改法：
  移到管理后台、开发模式或折叠到高级工具区，不放在主用户流里。

##### P1 每张文档卡都重复解释“还没有生成卡片”

- 归类：`文案问题`
- 影响：
  列表中每张卡都出现 `还没有生成卡片 / 可以直接从这份资料生成卡片`，重复感很强。
- 建议改法：
  状态收缩为一行：
  `未生成卡片`
  按钮直接写 `生成卡片`。

##### P1 头部、拖拽区、筛选区都在讲同一件事

- 归类：`UI+文案问题`
- 影响：
  页面上半区同时承担“介绍页”“上传入口”“筛选入口”，节奏偏散。
- 建议改法：
  保留标题。
  上传区只讲上传。
  筛选区只讲筛选，不再额外重复知识库用途。
