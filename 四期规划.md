# ByteCoach 四期规划

> 四期核心目标：从"功能完备"升级为"生产就绪"——补齐安全短板、强化管理后台、实现容器化部署，让项目真正具备上线和多人协作的能力。

---

## 4.1 登录安全增强

**目标**：补齐认证体系的安全短板，支持设备管理、异常检测和两步验证。

### 4.1.1 登录设备管理（P1，1d）✅ 已完成

**后端**
- ✅ 新建 `login_device` 表（id, user_id, device_fingerprint, device_name, ip, city, last_active_time, status, create_time）
- ✅ 每次登录时自动记录设备信息（从 X-Forwarded-For / X-Real-IP 解析客户端 IP）
- ✅ 新建 `GET /api/auth/devices`：返回当前用户已登录设备列表（含 current 标记）
- ✅ 新建 `POST /api/auth/devices/{id}/revoke`：远程登出指定设备（Redis 黑名单 + DB status 标记）
- ✅ 新建 `POST /api/auth/devices/revoke-all`：登出除当前设备外的所有设备
- ✅ 登录响应增加 `deviceId` 字段，JWT 内嵌 `deviceId` + `dfp`（设备指纹）声明
- ✅ JwtAuthenticationFilter 增加设备撤销检查（Redis device:token 黑名单）

**前端**
- ✅ 新建 `DeviceManagePage`：设备列表（设备图标、设备名、IP、城市、最后活跃时间），支持单个撤销和全部撤销
- ✅ 新建 `utils/device.ts`：浏览器指纹生成 + 设备名解析 + localStorage 持久化
- ✅ 登录/注册自动携带 deviceFingerprint + deviceName
- ✅ HTTP 拦截器自动附加 X-Device-Id 请求头
- ✅ AppShellHeader 用户区域新增"设备管理"入口链接
- ✅ 全局搜索（Cmd+K）新增"已登录设备"选项

### 4.1.2 登录异常检测（P2，1d）✅ 已完成

**后端**
- ✅ 新建 `login_log` 表（id, user_id, ip, city, device, status, fail_reason, create_time）
- ✅ 每次登录（成功/失败）写入 `login_log`
- ✅ IP 地理位置解析：调用免费 IP API（ip-api.com）
- ✅ 异地登录检测：对比上次登录城市，不同则发送通知
- ✅ 连续失败锁定：同一账号 5 分钟内连续失败 5 次，自动锁定 30 分钟
- ✅ 新建 `GET /api/auth/login-logs`：分页查询登录日志
- ✅ 管理后台 `GET /api/admin/login-logs`：管理员查看所有用户登录日志
- ✅ 新建 `GET /api/auth/captcha`：图形验证码生成接口

**前端**
- ✅ 个人设置页新增"登录历史"Tab（时间、IP、城市、状态、设备）
- ✅ 登录页增加图形验证码（连续失败 3 次后触发）
- ✅ 管理后台新增"登录日志"Tab
- ✅ 路由重构：`/settings/devices` → `/settings`（合并设备管理与登录历史）

### 4.1.3 两步验证 TOTP（P2，2d）✅ 已完成

**后端**
- ✅ 扩展 `user` 表：新增 `totp_secret VARCHAR(64)`、`totp_enabled BOOLEAN DEFAULT FALSE`、`recovery_codes TEXT`（JSON 数组，AES 加密存储）
- ✅ 新建 `POST /api/auth/2fa/setup`：生成 TOTP secret + QR code URL（otpauth:// 格式）
- ✅ 新建 `POST /api/auth/2fa/enable`：验证首次 TOTP code 后启用，返回 recovery codes
- ✅ 新建 `POST /api/auth/2fa/disable`：验证当前 TOTP code 后关闭
- ✅ 修改登录流程：如果用户启用了 2FA，登录接口返回 `requires2fa: true` + `tempToken`，前端跳转到验证码输入页
- ✅ 新建 `POST /api/auth/2fa/verify`：验证 TOTP code 或 recovery code，通过后返回正式 JWT
- ✅ Recovery code 使用后标记为已用，全部用完时提示重新生成
- ✅ 引入 `dev.samstevens.totp:totp:2.7.1` 库处理 TOTP 生成与验证

**前端**
- ✅ 新建 `TwoFactorSetupPage`：展示 QR code、输入验证码确认、展示 recovery codes（可复制/下载）
- ✅ 新建 `TwoFactorVerifyPage`：登录后跳转到此页输入 6 位验证码或恢复码
- ✅ 个人设置页新增"两步验证"Tab（状态显示 + 启用/禁用按钮）

## 4.2 管理后台增强

**目标**：将管理后台从"内容管理"扩展为"用户管理 + 系统监控"的完整运营工具。

### 4.2.1 用户管理（P1，1d）

**后端**
- 新建 `GET /api/admin/users`：分页查询用户列表（支持搜索、按角色筛选）
- 新建 `PUT /api/admin/users/{id}`：修改用户信息（昵称、角色、状态）
- 新建 `POST /api/admin/users/{id}/ban`：封禁用户（设置 status=disabled，踢出在线设备）
- 新建 `POST /api/admin/users/{id}/unban`：解封用户
- 新建 `GET /api/admin/users/{id}/detail`：用户详情（注册时间、最后登录、学习统计）

**前端**
- 管理后台新增"用户管理"Tab
- 用户列表：头像、用户名、昵称、角色、状态、注册时间、最后登录
- 操作：编辑角色、封禁/解封、查看详情

### 4.2.2 内容审核（P1，1d）

**后端**
- 扩展 `community_question` 和 `community_answer` 表：新增 `status` 字段（pending / approved / rejected）
- 新建 `GET /api/admin/community/pending`：待审核内容列表
- 新建 `POST /api/admin/community/{id}/approve`：审核通过
- 新建 `POST /api/admin/community/{id}/reject`：审核拒绝（附带原因）
- 社区内容发布后默认 status=pending，审核通过后才对外可见

**前端**
- 管理后台新增"内容审核"Tab
- 待审核列表：标题/内容预览、作者、发布时间、操作按钮
- 审核拒绝时弹窗填写原因

### 4.2.3 系统概览（P2，1d）

**后端**
- 新建 `GET /api/admin/overview`：系统统计数据（总用户数、今日活跃、今日新增、总面试次数、总复习次数）
- 新建 `GET /api/admin/overview/trend`：近 30 天用户增长和活跃趋势

**前端**
- 管理后台首页展示系统概览卡片：总用户、今日活跃、今日新增、总面试次数
- 用户增长折线图（ECharts）

## 技术风险

- **TOTP**：需要安全存储 totp_secret（建议 AES 加密），recovery codes 也需要加密存储。时钟偏移 ±30s 容忍窗口
- **IP 地理位置**：GeoLite2 需要定期更新（每月），免费 API 有调用频率限制
- **批量导入**：大文件上传需要异步处理（返回任务 ID + 轮询结果），避免阻塞请求
- **Docker**：MySQL 和 Redis 数据卷需要持久化，避免容器重建丢失数据
